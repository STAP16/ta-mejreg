# Отчет модуль А:
## Архитектура ПО и таблиц

### Анализ функциональных и нефункциональных требований:

Функциональные требования: 
1) Пакетный пересчет исторических метрик
2) Захват видеопотока
3) Детекция и классфикация т/c 
4) Трекинг объектов между кадрами
5) Вывод данных в аналитический дашборд

Нефункциональные:
1) Realtime обновление
2) Запуск программы одной командой
3) Единая версия правды
4) Масштабируемость — добавление новых камер без перестройки архитектуры
5) Хранение полной истории за 180 дней

### Архитектура DWH
Для построение DWH выбрана медальонная архитектура (bronze -> silver -> gold)
bronze слой: Содержит в себе сырые данные, не очищая и не удаляя их, обеспечевая полный контроль за исходными данными
silver слой: Является Source Of Truth для gold слоя. В данном слое происходит агрегация и очистка данных. Подготовка их к витринному слою
gold слой: Все витрины храняться в данном, источником данных для всех витрин является silver слой таблица: silver_detections

#### Описание витрин и их частота обновление
Источником данных для всех витрин является silver слой таблица: silver_detections

1) Витрина: v_current_load
	Бизнес вопросы на который отвечает витрина: Какой сейчас траффик в реальном времени, Какие данные по сравнению с прошлыми периодами
	Частота обновления: 1 минуты
	Обоснование частоты обновления: Данная витрина является траффиком в реальном времени, требует свежие данные для диспетчера

2) Витрина: v_forecast_30_60_120
	Бизнес вопросы на который отвечает витрина: Какова текущая загруженность на горизонтах 30/60/120 мин
	Частота обновления: 30 минут
	Обоснование частоты обновления: Прогнозы саставляются каждые 30 минут на 30/60/120 минут вперед, 
	обновление витрины каждые 30 минут позволяет собрать данные за последние 30 минут и на их основе сделать новый прогноз. 

3) Витрина: v_heavy_truck_patterns
	Бизнес вопросы на который отвечает витрина: Паттерны появляение большегрузного транспорта в часы пик
	Частота обновления: Каждый час
	Обоснование частоты обновления: На основе данных за последний час, можно наиболее точно выявить паттерн появляение большегрузного транспорта.

4) Витрина: v_vehicle_structure
	Бизнес вопросы на который отвечает витрина: Какова структура транспортного потока
	Частота обновления: Каждую минуту
	Обоснование частоты обновления: Данная витрина неотемлемая часть realtime обновления, Диспетчер отслеживает структуру потока в реальном времени

### Стек и его обоснование
1) ClickHouse - Колоночная СУБД, поддерживает партиции по датам, движок MergeTree, благодаря своей структуре позволяет делать быстрые аналитические запросы / выборки.
2) YOLO - Предобученная модель на датасете COCO, поддерживает встроеный трекинг объектом между кадрами, предоставляет файлы весов и конфигурации
3) Apache Airflow - Стандарт индустрии для построение DAG, используется для ETL/ELT пайпланов
4) Metabase - Аналитический дашборд, поддерживает realtime обновление
5) Docker - контейнерезация, обеспичивает запуск всей платформы одной командой.
6) Apache Kafka - Брокер сообщений, Producer / Consumer, Запись детекций в отдельный TOPIC, необходим, чтобы предотвращать потерю данных при сбоях.
7) MinIO S3 - хранилеще для обработанных кадров
8) ЯП: Python - Поддерживает документацию со всеми вышеперичсленными библиотеками, стандарт индустрии для Аналитика.
